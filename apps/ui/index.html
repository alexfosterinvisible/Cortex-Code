Classification
/classify_issue Utility Classify GitHub issue → /chore, /bug, /feature, /patch
▼
Purpose: Classify a GitHub issue into the appropriate workflow type.

Command Mapping

  • /chore – Maintenance, refactoring, dependency updates
  • /bug – Bug fixes, error corrections
  • /feature – New features, enhancements
  • /patch – Quick patches for review issues
  • 0 – Unclassifiable

Arguments

$ARGUMENTS = GitHub issue JSON (number, title, body)

Recognized Commands

📋 Planning Phase
/feature Plan Create feature implementation plan
▼
Purpose: Create a comprehensive implementation plan for a new feature.

Variables

  • $1 = issue_number
  • $2 = adw_id
  • $3 = issue_json

Plan Format Sections

  • Metadata (issue_number, adw_id)
  • Feature Description
  • User Story (As a... I want... So that...)
  • Problem Statement
  • Solution Statement
  • Relevant Files
  • Implementation Plan (Phase 1-3)
  • Step by Step Tasks
  • Testing Strategy
  • Acceptance Criteria
  • Validation Commands

Output

specs/issue-{issue_number}-adw-{adw_id}-sdlc_planner-{name}.md
/bug Plan Create bug fix plan
▼
Purpose: Create a thorough plan to fix a bug at the root cause level.

Plan Format Sections

  • Bug Description
  • Problem Statement
  • Solution Statement
  • Steps to Reproduce
  • Root Cause Analysis
  • Step by Step Tasks
  • Validation Commands

Key Instruction

"Be surgical with your bug fix, solve the bug at hand and don't fall off track."

▼
Purpose: Create a simple but thorough plan for maintenance tasks.

Plan Format Sections

  • Chore Description
  • Relevant Files
  • Step by Step Tasks
  • Validation Commands

▼

Available Workflows

ADW orchestrates Claude agents through git worktrees for parallel, isolated execution

📋
adw_plan_iso
Planning only

Fetches GitHub issue → classifies → creates worktree → generates implementation plan → creates PR

uv run adw plan 42
🔨
adw_build_iso
Building only (requires plan)

Executes /implement with existing plan file. Requires prior plan phase.

uv run adw build 42 abc12345
🧪
adw_test_iso
Testing with auto-fix iterations

Unit Tests: Up to 4 iterations (run → fix failures → re-run)

E2E Tests: Up to 2 iterations via Playwright MCP

Flow: Run tests → Parse failures → Fix each SERIALLY → Re-run all → Repeat until pass or max attempts

Stop when: All pass, OR max iterations, OR no fixes made (no point retrying)

uv run adw test 42 abc12345 --skip-e2e
🔍
adw_review_iso
Review with blocker-fix iterations

Up to 3 iterations: Review → Fix blockers → Re-review

Flow: Review against spec → Screenshot UI → Classify issues by severity → Fix blockers SERIALLY → Re-review

Only blockers auto-fixed — tech_debt and skippable issues are noted but not resolved

uv run adw review 42 abc12345 --skip-resolution
📖
adw_document_iso
Documentation only

Generates markdown documentation from git diff and spec file.

uv run adw document 42 abc12345
🚀
adw_ship_iso
Ship/merge PR

Approves and merges the pull request to main branch.

uv run adw ship 42 abc12345

Test & Review Iteration Logic
🧪
Test Phase Iterations
Serial fix → Re-run all
MAX_TEST_RETRY_ATTEMPTS = 4  (unit tests)
MAX_E2E_TEST_RETRY_ATTEMPTS = 2  (browser tests)

for attempt in 1..max_attempts:
  results = run_all_tests()
  if all_passed: break ✓
  
  failed = [t for t in results if !t.passed]
  resolved = 0
  for test in failed:  # SERIAL
    if resolve_test(test): resolved++
  
  if resolved == 0: break ✗  # No point retrying

Why serial? Fixes may have cascading effects. Fixing test A might automatically fix test B.

🔍
Review Phase Iterations
Only blockers → Re-review
MAX_REVIEW_RETRY_ATTEMPTS = 3

for attempt in 1..max_attempts:
  result = review_against_spec()
  blockers = [i for i in result.issues 
              if i.severity == "blocker"]
  
  if !blockers: break ✓  # tech_debt/skippable OK
  
  for blocker in blockers:  # SERIAL
    plan = create_patch_plan(blocker)
    implement(plan)

Why serial? Patches may conflict. Each fix is committed before the next to maintain stable state.

Review Issue Severity Rubric
Severity Auto-Fixed? Criteria Examples
blocker ✓ Yes Breaks core functionality, security issue, or explicit requirement not met • Button doesn't submit form
• Auth bypass vulnerability
• "Must have dark mode" but no toggle
• Data not persisting to DB
tech_debt ✗ No (logged) Works but suboptimal. Should fix eventually but doesn't block release. • Hardcoded values that should be config
• Missing error boundary
• N+1 query pattern
• Inconsistent naming
skippable ✗ No (noted) Nice-to-have, style preference, or outside scope • Could use memo() for perf
• Prefer different icon
• Suggestion for future feature
• Minor UI alignment
Model Selection

ADW selects models based on model_set in state:

Model Set Commands
Heavy (Opus) /implement, /document, /feature, /bug, /chore, /patch
Base (Sonnet) Everything else (classify, review, commit, etc.)

CLI Commands

Run ADW workflows from the command line

# Install ADW in your project
uv add ../adw-framework

# Verify installation
uv run adw --help
Command Arguments Options Description
adw plan issue_number [adw_id] - Plan implementation for an issue
adw build issue_number adw_id - Build implementation from plan
adw test issue_number adw_id --skip-e2e Run tests with auto-fix
adw review issue_number adw_id --skip-resolution Review implementation against spec
adw document issue_number adw_id - Generate documentation
adw ship issue_number adw_id - Ship changes (merge PR)
adw patch issue_number [adw_id] - Create and implement a patch
adw sdlc issue_number [adw_id] --skip-e2e --skip-resolution Run complete SDLC pipeline
adw zte issue_number [adw_id] --skip-e2e --skip-resolution Zero Touch Execution (SDLC + Ship)
adw monitor - - Run cron monitor for new issues
adw webhook - - Run webhook server
Webhook Triggers

Comment on a GitHub issue to trigger workflows: 

# Trigger planning only
adw_plan_iso

# Trigger full SDLC
adw_sdlc_iso

# Trigger ZTE (must be capitalized!)
adw_sdlc_ZTE_iso

# Use heavy models for complex tasks
adw_sdlc_iso model_set heavy

# Resume existing workflow
adw_build_iso adw_id abc12345

Configuration

Configure ADW for your project

.adw.yaml
# .adw.yaml - ADW Framework Configuration

# REQUIRED: Project identification (usually matches GitHub org/repo)
project_id: "your-org/your-repo"

# PATHS
artifacts_dir: "./artifacts"      # State, logs, worktrees
source_root: "./src"             # Base path for new code

# PORTS - For isolated worktrees (supports 15 concurrent agents)
ports:
  backend_start: 9100
  backend_count: 15
  frontend_start: 9200
  frontend_count: 15

# COMMANDS - Claude command paths (framework + project overrides)
commands:
  - "${ADW_FRAMEWORK}/commands"   # Built-in ADW commands
  - ".claude/commands"            # Project-specific overrides

# APP CONFIG - Project-specific settings (customize for your stack)
app:
  # Directory structure
  backend_dir: "app/server"
  frontend_dir: "app/client"

  # Scripts
  start_script: "scripts/start.sh"
  stop_script: "scripts/stop.sh"
  reset_db_script: "scripts/reset_db.sh"

  # Commands
  test_command: "uv run pytest"
  lint_command: "uv run ruff check"
  build_command: "uv run build"
.env
# Required
ANTHROPIC_API_KEY=sk-ant-xxx

# Optional (uses `gh auth` if not set)
GITHUB_PAT=ghp_xxx

# Optional (defaults to "claude")
CLAUDE_CODE_PATH=claude

└── trees/{adw-id}/                # Isolated git worktree
ADW ID

8-character unique identifier per workflow instance (e.g., abc12345). Used for:

  • State file location: artifacts/{project_id}/{adw-id}/adw_state.json
  • Worktree path: artifacts/{project_id}/trees/{adw-id}/
  • Deterministic port allocation (hash-based)
